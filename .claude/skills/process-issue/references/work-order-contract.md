# 업무 지시서 계약 (work-order contract)

설계자가 개발자에게 넘기는 **모호함 없는 목표 명세**의 표준 형식. 개발자가 스스로 추측하거나
결정하지 않도록, 이 형식의 모든 칸을 채워서 전달한다. 채울 수 없는 칸이 있으면 그건 설계자가
아직 이슈를 충분히 이해하지 못했다는 신호다 — 개발 착수 전에 사용자에게 되묻는다.

## 형식

```markdown
## 업무 지시서: <조각 이름> (이슈 #<n>)

### 목표 (한 문장)
<이 조각이 끝나면 무엇이 가능해지는가. 구현 방법이 아니라 결과로 쓴다.>

### 왜 (맥락)
<이슈의 어떤 요구에서 나왔는가. 관련 코드/파일 포인터.>

### 범위 — 대상 파일
- <손대도 되는 파일 목록. 다른 조각과 겹치지 않아야 한다.>

### 수용 기준 (검증 가능하게)
1. <입력 → 기대 출력 / 실행 명령 → 통과 조건. 테스터가 그대로 확인할 수 있어야.>
2. ...

### 비목표 (하지 말 것)
- <범위 밖. 스코프 크립 차단. 예: "UI 문구는 이번에 바꾸지 않는다".>

### 검증 힌트
- <어떤 테스트/실행으로 확인하는지. 예: `pytest tests/test_x.py::test_y`, 라우트 `GET /api/z` 호출.>

### 의존
- <선행 조각이 있으면 명시. 없으면 "없음".>
```

## 좋은 목표 / 나쁜 목표

| 나쁨 (모호) | 좋음 (검증 가능) |
|------------|-----------------|
| "유형 삭제 기능 개선" | "`DELETE /api/types/{slug}` 호출 시 해당 유형 문서가 `etc`로 이동하고 204를 반환한다. `test_api_delete_moves_docs_to_etc`로 확인." |
| "에러 처리 추가" | "중복 슬러그로 `POST /api/types` 시 409와 `{detail}`를 반환한다(500 아님)." |
| "테스트 보강" | "`rename_type`의 etc 보호 경로에 대한 테스트를 추가해 `etc` 이름변경 시도가 거부됨을 덮는다." |

## 분해 기준 (복잡 이슈)

- **파일 경계로 나눈다.** 조각 A와 B가 같은 파일을 고치면 병렬이 충돌한다. 겹치면 순차로
  묶거나 worktree 격리.
- **딜리버러블로 나눈다.** 이슈가 "추가 + 이름변경 + 삭제"처럼 여러 기능이면 기능당 한 조각.
- **수직 슬라이스 선호.** "모델만" "API만"보다 "이 기능의 모델+API+테스트"가 한 조각이면 검증이 닫힌다.
- **조각은 독립 검증 가능해야 한다.** 테스터가 다른 조각을 기다리지 않고 그 조각만으로 PASS/FAIL을 낼 수 있게.
