# 검증 플레이북 (테스터용)

테스터는 "테스트가 통과한다"가 아니라 **"설계자의 수용 기준이 실제로 달성됐다"**를 증거로
보인다. 단위테스트와 실제 실행 두 층으로 확인한다.

## 1. 수용 기준 대조 (기준이 곧 체크리스트)

업무 지시서의 수용 기준 각 항목을 그대로 가져와 PASS/FAIL 표로 만든다. 개발자의 완료 보고를
믿지 말고, 각 항목을 직접 재현한다.

```markdown
## 판정: <조각> (이슈 #<n>)
| # | 수용 기준 | 판정 | 근거(명령·출력) |
|---|-----------|------|----------------|
| 1 | DELETE 시 문서가 etc로 이동 | PASS | `pytest ...::test_delete_moves` → 1 passed |
| 2 | 중복 POST → 409 | FAIL | 실제 500 반환. 아래 재현 참조 |
```

## 2. 단위 검증 (pytest)

- 이 repo는 `.venv/bin/python -m pytest`로 돈다. 관련 파일만 좁혀 실행:
  `.venv/bin/python -m pytest tests/test_doctypes.py -q`.
- 수용 기준을 덮는 테스트가 **없으면 보강**한다. 단, `tests/` 하위에만 추가하고 구현은 고치지 않는다.
- 전체 회귀도 한 번 돌려 다른 곳이 깨지지 않았는지 본다.

## 3. 동작 검증 (실제 실행)

단위테스트만으로 "정말 동작하는가"가 안 잡히면 실제로 굴린다.

- **서버 라우트**: `TestClient`로 호출하거나, 필요 시 로컬 기동 후 `curl`. 인증이 필요한 라우트는
  `REPORTS_UPLOAD_USER/PASS`를 쓴다. 존재하지 않는 경로도 함께 때려 미들웨어가 라우팅 전에
  가로채는지 확인한다(라우트 존재 증명을 401/404로 착각하지 않도록 — 컨테이너/모듈 내부를 직접 본다).
- **스크립트**: `DRY_RUN=1` 등 부작용 없는 모드로 먼저 확인.
- **경계면 교차 확인**: API 응답 shape ↔ 이를 소비하는 프론트/호출부를 **동시에 읽어** 필드명·타입이
  맞는지 본다. "양쪽 파일이 존재한다"가 아니라 "양쪽이 같은 계약을 말한다"를 확인.

## 4. 증거 규칙

- 통과를 적기 전에 **명령을 실제로 실행**하고 출력을 붙인다. 실행하지 않은 것을 PASS로 적지 않는다.
- 검증 못 한 항목(환경 불가 등)은 FAIL이 아니라 **"미검증"**으로 분리 표기하고 이유를 남긴다.
  보고자가 이를 PR에 그대로 옮긴다(통과로 위장하면 안 됨).

## 5. FAIL 반려 형식

개발자가 바로 고칠 수 있도록 구체적으로:

```
FAIL 기준 #2 (중복 POST → 409)
- 입력: POST /api/types {slug: "demo"} (이미 존재)
- 기대: 409 + {detail}
- 실제: 500 IntegrityError (스택: ...)
- 지목: server.py 의 create_type 예외 처리 누락으로 추정
```
