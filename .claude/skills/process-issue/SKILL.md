# 이슈 처리 팀 (process-issue)

GitHub 이슈 하나를 받아 **개발 → 검증 → PR** 까지 끝내는 오케스트레이터다. 네 역할이 협업한다:
**설계자**(파악·분해·조율·책임) · **개발자**(구현) · **테스터**(검증) · **보고자**(PR).

**메인 세션이 설계자(리더)를 맡는다.** 개발자·테스터·보고자는 `Agent` 도구로 스폰하고,
공유 작업 보드(`TaskCreate`)와 `SendMessage`로 조율한다. 각 역할 정의는 `.claude/agents/issue-*.md`.

## 언제 / 언제 아닌가

- **사용**: "6번 이슈 처리해줘", "이슈 #3 개발해서 PR 올려줘", "이 이슈 반영", 그리고 후속("다시", "이어서", "수정").
- **사용 안 함**:
  - 이슈를 **조회·요약만** 원할 때 → 파이프라인 없이 직접 답한다(`gh issue view`).
  - **완성된 HTML 리포트 등록** → `register-report` 스킬(별개 하네스).

## 입력

- **이슈 번호(필수).** 없으면 되묻는다. 저장소는 기본 현재 repo(`CREFLEINC/reports`).
- 끝점은 **PR 생성**이다. 자동 머지하지 않는다(사람 리뷰 몫).

## 모델 배정

| 역할 | 에이전트 | 모델 | subagent_type |
|------|----------|------|---------------|
| 설계자 | (메인 세션) `issue-architect` | Opus 4.8 | — |
| 개발자 | `issue-developer` | **opus** | general-purpose |
| 테스터 | `issue-tester` | **sonnet** | general-purpose |
| 보고자 | `issue-reporter` | **sonnet** | general-purpose |

> 사용자 지정 배정이다. 하네스 기본값("전부 opus")을 의도적으로 오버라이드한다. `Agent` 호출마다
> `model` 을 위 표대로 명시한다. 검증 스크립트 실행이 필요하므로 서브타입은 `general-purpose`
> (`Explore`는 읽기 전용이라 부적합).

## Phase 0: 컨텍스트 확인

`_workspace/issue-<n>/` 존재 여부로 실행 모드를 가른다.

- 없음 → **초기 실행** (아래 Phase 1부터).
- 있음 + 사용자가 부분 수정 요청 → **부분 재실행** (해당 조각의 개발자/테스터만 재호출).
- 있음 + 새 방향 지시 → 기존을 `_workspace/issue-<n>_prev/`로 옮기고 **새 실행**.

## Phase 1: 설계자 트리아지 & 모드 선택

메인 세션(설계자)이 수행한다.

1. **착수**: `scripts/start_issue.sh <n>` — 이슈 조회 + 최신 `origin/main` 기준 작업 브랜치 생성.
   본문·코멘트·연결 PR을 읽는다. 관련 코드도 확인한다.
2. **수용 기준 도출**: 이슈가 "무엇이 되면 끝인가"를 검증 가능한 항목으로 뽑는다. 모호하면
   착수 전에 사용자에게 되묻는다(추측 금지).
3. **모드 선택** — 아래 루브릭:

   | 신호 | 서브에이전트(단순) | 에이전트 팀(복잡) |
   |------|------------------|-----------------|
   | 변경 파일 수 | ~1–2 | 3+ 또는 여러 모듈 |
   | 딜리버러블 수 | 1 | 2+ (이슈에 기능 여러 개) |
   | 난이도·위험 | 낮음(국소·명확) | 높음(런타임·CI·데이터 영향) |
   | 분해 필요성 | 불필요 | 필요 |

   - **문서/한 줄/설정** 수준(trivial): 설계자가 직접 처리하고 테스터만 검증.
   - **단순**: 서브에이전트 패스트패스(Phase 2A).
   - **복잡**: 에이전트 팀(Phase 2B).
4. **업무 지시서 작성**: 각 조각을 `references/work-order-contract.md` 형식으로 만든다.
   목표·범위·수용기준·대상파일·비목표·검증힌트를 못박아 개발자가 추측하지 않게 한다.
   복잡 이슈는 **파일이 겹치지 않는 조각**으로 나눈다.
5. 계획을 `_workspace/issue-<n>/01_architect_plan.md`에 저장한다.

## Phase 2A: 실행 — 서브에이전트 (단순)

순차 파이프라인. 팀 통신 오버헤드 없이 결과만 주고받는다.

1. **개발자**(opus) 스폰 — 업무 지시서를 프롬프트로 전달. 완료 보고를 받는다.
   ```
   Agent(subagent_type="general-purpose", model="opus",
         prompt="너는 issue-developer 다. .claude/agents/issue-developer.md 를 읽고 따르라.
                 업무 지시서: <work-order>. 산출물은 _workspace/issue-<n>/02_dev_main_report.md.")
   ```
2. **테스터**(sonnet) 스폰 — 수용 기준 + 개발자 보고를 전달, PASS/FAIL 판정을 받는다.
3. **검증 루프**(Phase 3)로 이동.

## Phase 2B: 실행 — 에이전트 팀 (복잡)

1. **작업 보드 구성**: 조각마다 `TaskCreate`(의존관계 포함).
2. **개발자 병렬 스폰**(opus, `run_in_background: true`): 각자 **겹치지 않는 파일 스코프**의
   조각 1건. 스코프가 겹칠 위험이면 순차로 조정하거나 `isolation: "worktree"`.
3. 각 개발자 완료 즉시 **테스터**(sonnet)가 그 조각을 **점진 검증**한다(전체 완성 후 1회 금지).
4. 설계자는 작업 보드로 상태를 추적하고, FAIL은 `SendMessage`로 원 개발자에게 되돌린다.
5. 모든 조각 PASS면 통합 상태에서 회귀 검증(전체 pytest) 후 Phase 4.

## Phase 3: 검증 루프

테스터는 `references/verification-playbook.md`의 방법론(수용 기준 대조 → 단위 → 실제 실행 → 증거)으로 판정한다.

- 테스터 판정이 **FAIL**: 결함(어떤 기준·어떤 입력·어떻게 어긋남)을 개발자에게 되돌려 수정 → 재검증.
- 같은 조각 **2회 연속 실패**: 설계자가 목표를 더 잘게 쪼개거나 접근을 바꿔 재지시. 3회면 사용자에게 판단 요청.
- 전체 조각 PASS + 회귀 없음 → Phase 4.

## Phase 4: 정리 & PR

1. **보고자**(sonnet) 스폰 — `_workspace/issue-<n>/`의 계획·개발·판정 산출물을 근거로:
   커밋 정리 → PR 본문 작성(`Closes #n`, 무엇을·왜, 조각별 변경, 검증 근거, 비목표) → `main` 기준 PR 생성.
2. **설계자 최종 검토**: 보고자가 반환한 PR이 이슈 수용 기준을 충족하는지 확인한다. 미흡하면
   보고자에게 보완을 지시한다. 충족하면 사용자에게 PR URL과 요약을 보고한다.

> "설계자가 내용을 정리하여 PR을 생성한다"는 요구의 해석: 보고자가 PR 작성·게시를 수행하고,
> 설계자가 전체를 총괄해 최종 PR을 성립시킨다(책임 소재는 설계자).

## 데이터 전달 프로토콜

- **작업 보드**(`TaskCreate`/`TaskUpdate`): 조각 상태·의존관계 추적(팀 모드).
- **파일 기반**(`_workspace/issue-<n>/`): 산출물. 컨벤션 `{순번}_{역할}_{조각}.md` — 예
  `01_architect_plan.md`, `02_dev_api_report.md`, `03_tester_api_verdict.md`, `04_reporter_pr.md`.
  중간 산출물은 보존(감사 추적), 코드는 작업 트리에 직접.
- **메시지**(`SendMessage`): 스폰된 에이전트를 컨텍스트 유지한 채 이어 호출(결함 반려·재작업).

## 에러 핸들링

- 에이전트 1회 재시도 후 재실패 → 해당 산출 없이 진행하되 최종 보고에 **누락을 명시**한다.
- 상충하는 결과는 삭제하지 않고 **출처를 병기**한다.
- 검증 환경 불가(내부망·docker 없음)로 테스터가 일부만 검증 → 판정서·PR에 **미검증 항목을 명시**한다(통과로 위장 금지).
- 이슈 목표가 끝내 모호 → 파이프라인을 멈추고 사용자에게 구체 질문.

## 테스트 시나리오

- **정상(단순)**: "이슈 #3 처리" → 트리아지=단순 → 개발자 1 → 테스터 PASS → 보고자 PR 생성 →
  설계자 검토 → URL 보고. `.deployed_sha`·서버 등 운영은 건드리지 않음(PR까지만).
- **정상(복잡)**: 다기능 이슈(예 #6류) → 조각 3개 분해 → 병렬 개발자(opus, 겹치지 않는 파일) →
  조각별 점진 검증 → 전체 회귀 → 통합 PR.
- **에러(검증 FAIL)**: 테스터가 수용기준 2번 FAIL → 결함을 개발자에 반려 → 수정 → 재검증 PASS → 진행.
- **에러(모호)**: 이슈에 수용 기준을 세울 수 없음 → 개발 착수 없이 사용자에게 질문.
