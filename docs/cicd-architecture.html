요소별 설명

다이어그램의 각 구성 요소가 하는 일.

GitHub 클라우드

트리거 schedule + dispatch

매일 0 8 * * *(UTC = 17:00 KST) 예약 실행과, 필요할 때 누르는 수동 workflow_dispatch. push·PR 트리거는 두지 않아, 신뢰할 수 없는 코드가 사내 러너에서 실행되는 것을 원천 차단한다.

daily-deploy 워크플로

.github/workflows/deploy.yml. runs-on: [self-hosted, hulk]로 hulk 러너를 지정하고, 전체 이력을 checkout한 뒤 ops/deploy.sh를 호출하는 얇은 껍데기. 동시 실행 방지(concurrency) 가드 포함.

hulk 러너 · ops/deploy.sh

self-hosted runner

hulk(192.168.1.111) 안에서 도는 GitHub 러너. sudo 없이 systemd --user + linger로 상시화되어 재부팅·크래시에도 자동 복구. GitHub 클라우드가 못 들어오는 사내망 자원에 로컬로 접근한다.

① main 변경 있나?

배포 디렉터리의 .deployed_sha 마커와 현재 main HEAD를 비교. 같으면 즉시 skip(종료), 다르면 진행. 하루 여러 커밋도 한 번에, 걸린 날은 다음 실행에서 보정하는 멱등 방식.

② 경로 분류 (git diff)

git diff로 바뀐 파일을 분류 — proposals/만 바뀌면 rsync만, server.py·Dockerfile 등 코드면 뷰어 이미지 재빌드, 렌더러 파일이면 렌더러 재빌드.

③ pytest 게이트 (3.12)

배포 전 python:3.12-slim 컨테이너에서 pytest 실행(hulk 시스템 파이썬은 3.8이라 운영과 동일 환경을 컨테이너로 확보). 하나라도 실패하면 배포 중단.

④ 조건부 실행

분류 결과에 따라 콘텐츠 rsync / 이미지 빌드·push / compose 배포를 선택 수행. 빌드는 immutable :<short-sha> 태그로 Harbor에 push하고 .env의 태그 변수를 갱신한다.

⑤ healthz 정상?

배포 후 /healthz를 폴링. 정상이면 통과, 실패하면 이전 태그로 자동 롤백하고 잡을 실패시켜 GitHub 이메일로 알린다. 마커는 갱신하지 않는다.

⑥ deployed_sha 기록

전 과정이 성공했을 때만 마커를 현재 HEAD로 전진. 이 한 줄이 “다음엔 여기부터”를 정의해 다음 실행의 변경 판정 기준이 된다.

사내망 LAN 자원

Harbor 레지스트리

hub.crefle.com. 재빌드된 뷰어/렌더러 이미지를 short-SHA 태그로 보관. compose가 여기서 pull해 배포한다. 사내망 전용이라 클라우드 러너로는 push 불가.

proposals 마운트

큐레이션 문서의 소스. 코드 재빌드 없이 rsync로 동기화되며 컨테이너에 :ro bind mount로 즉시 반영(매 요청 스캔이라 재시작 불필요).

reporter + renderer (:28080)

docker compose로 뜨는 운영 서비스 — 뷰어(업로드 API 포함)와 격리 렌더러. compose는 이미지 태그를 ${REPORTER_TAG}/${RENDERER_TAG} 변수로 참조해 롤백·재현이 쉽다.

동작 흐름 (세 갈래)

핵심 설계 결정

왜 이렇게 만들었나.

사내망 접근 → self-hosted runner
hulk(192.168.1.111)와 Harbor(hub.crefle.com)는 사설 LAN 전용이라 GitHub 클라우드 러너로 SSH·rsync·push가 불가능. 러너를 LAN 안에 두어 SSH·터널·외부 노출 없이 로컬로 빌드·배포한다. 덤으로 hulk가 x86_64라 크로스빌드도 사라진다.
변경 판정은 커밋 SHA 마커로
“지난 배포 이후 바뀌었나”를 .deployed_sha vs HEAD로 판단. 하루 여러 커밋을 한 번에, 러너가 꺼져 하루 걸러도 다음 실행에서 누락분까지, 재실행해도 안전한 멱등 동작.
테스트는 3.12 컨테이너에서
hulk 시스템 파이썬은 3.8이지만 서비스는 3.12 타깃. 게이트를 python:3.12-slim 컨테이너에서 돌려 운영과 동일 환경으로 검증한다.
immutable SHA 태그 + 자동 롤백
빌드는 short-SHA로 push하고 compose는 태그 변수를 참조. 배포 후 /healthz가 실패하면 이전 태그로 되돌려 재기동하고 마커를 전진시키지 않는다 — 깨진 배포가 남지 않는다.
운영 함정 기록 · sudo 무암호가 안 돼 공식 시스템 서비스(svc.sh) 대신 무루트 systemd --user + linger로 상시화. 그리고 gh 토큰에 workflow 스코프가 없어 워크플로 파일 push가 막혀, origin remote를 HTTPS→SSH로 전환해 우회했다.
